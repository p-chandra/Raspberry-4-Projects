# SETUP
The VMK180 can connect to a host machine, but the host computer is usually used for development, programming, and debugging, not for normal runtime operation.

Host Machine Usage
| Purpose              | What the host does    |
| -------------------- | --------------------- |
| Programming          | Load FPGA bitstream   |
| Boot loading         | Load OS / firmware    |
| Debugging            | Use JTAG debugger     |
| Logging              | View console output   |
| Software development | Build and upload code |

**Development Setup**
```
Developer PC
     │
     ├── JTAG → program FPGA
     ├── UART → serial console
     └── Ethernet → network/debug
           │
           ▼
      VMK180 Board
```

**Runtime in Real System**
```
Sensors
   │
   ▼
VMK180
   ├─ CPU running OS (Linux or ARINC 653)
   ├─ FPGA processing
   └─ I/O interfaces
```

**Boot Process**
```
Power on
   ↓
Boot ROM
   ↓
Bootloader
   ↓
FPGA bitstream loaded
   ↓
Operating system starts
   ↓
Applications / partitions run
```
The boot files may come from:
- QSPI flash
- SD card
- Network Boot
- or the host during development.

# Hypervisor vs RTOS
You do not need a hypervisor to run an RTOS on VMK180.
The RTOS can run directly on the hardware.
**Hypervisors are only needed if you want multiple operating systems sharing the same CPU resources.**

## RTOS Running Directly On Hardware
In many embedded systems, the RTOS runs directly on the CPU cores.

Architecture:
```
Hardware (Versal SoC)
   ↓
RTOS Kernel
   ↓
Tasks / Processes
```
Examples of RTOS that could run this way include:
- FreeRTOS
- VxWorks
- INTEGRITY
- Deos

The RTOS kernel handles:
- task scheduling
- interrupts
- memory management
- hardware drivers

No hypervisor is involved.

## When a Hypervisor Is Used
A hypervisor is used when you want multiple operating systems running at the same time.

Architecture:
```
Hardware
   ↓
Hypervisor
   ├─ RTOS
   ├─ Linux
   └─ Another RTOS
```
Reason to use a hypervisor

| Reason                   | Example                               |
| ------------------------ | ------------------------------------- |
| Mixed criticality        | Linux for UI + RTOS for control       |
| Isolation                | Safety-critical vs non-critical tasks |
| Multiple OS environments | Legacy + new software                 |
